The Flaky Network

A real-time UDP sender/receiver that carries a 160-byte frame every 20 ms across a lossy, jittery, reordering, duplicating relay — losing ≤ 1% of deadlines at ≤ 2× bandwidth, at the lowest playout delay the statistics allow.

C11, POSIX sockets only Reed-Solomon 4-of-7 erasure code share diversity: miss ≈ 35·f⁴ share-targeted ARQ no jitter buffer 1.88× proactive, ≤ 2.0× total

1. The problem, reduced to its physics

The source hands the sender frame i at exactly t0 + i·20ms — you can never send a frame early, because you don't have it yet. The player accepts frame i only if a correct copy arrives before a fixed deadline t0 + DELAY_MS + i·20ms. The score: miss rate ≤ 1%, relay bytes ≤ 2.0× the raw stream (both directions, counting drops), and — among valid runs — lowest DELAY_MS wins.

A lost frame has exactly two remedies: send it again (ARQ — request and reply each cross the hostile net, a full round trip) or send it ahead of time (proactive redundancy, bounded by the 2× budget). At 20 ms/frame the round trip is expensive, so proactive redundancy carries recovery; ARQ is a backstop for when the delay budget still affords a round trip.

The 16-bit wire seq is expanded RTP-style at the receiver (reordering is bounded by relay delay < 1 s ≈ 50 frames ≪ 32768); harness legs keep their fixed 4-byte big-endian seq. The scorer verifies every payload's SHA-256, so the job is to move bytes, never to guess them.

3. The core idea: many small independent draws beat two big ones

The relay draws an independent delay and drop for every packet it receives. Version 1 of this design exploited that with an immediate twin — two full copies, P(miss)=f² — but two 164 B copies cost 2.05×, so 1-in-14 frames went out unprotected, and at the floor those singles dominate the miss budget (f/14 ≫ f²). That pinned the minimum delay at the network's max one-way delay (dmax): 40 ms on profile A, 80 ms on B.

Reed-Solomon share diversity

Split each 160 B frame into K=4 data shares of 40 B, add P=3 parity shares from a Cauchy matrix over GF(256), and send 7 packets of 43 B at the same instant: 301 B/frame = 1.88× — every frame fully protected, budget to spare. The code is MDS: any 4 of the 7 shares reconstruct the frame.

twin      :  P(miss) = f²                 (f = per-packet loss-or-late prob)
RS(7,4)   :  P(miss) = P(≥4 of 7 fail) ≈ 35·f⁴

The frame survives a per-packet failure rate of ~13% instead of ~7% — and since "failure" includes lateness, the valid delay drops below dmax, which no full-copy scheme can do (it needs 4 independently-late draws at once). It also absorbs a 3-packet loss burst for free.

Correct by construction

A decode bug would forward wrong bytes — a silent scored miss. So both binaries run an exhaustive self-test at startup: every subset of ≥ 4 shares, for every parity count, must reconstruct pseudo-random payloads exactly, or the process aborts before joining the run. The Gauss-Jordan decode path is therefore proven over all 35 reconstruction patterns every single launch.

4. The receiver: reassemble, forward-now, clocked ARQ

No jitter buffer — on purpose

The deadline is externally fixed, so buffering is pure added latency. The receiver forwards a frame the instant it is reconstructable (4th share in, one GF(256) solve, ~µs) — exactly once, dedupe by sequence bitset.

Share-targeted, clock-triggered ARQ

Frame i is due at t0 + i·20ms, so "still missing at t_i + 85ms" is a clock decision — no waiting for later packets to reveal a gap. The NACK carries a bitmask of shares already held; the sender resends only the missing ones plus one spare (43 B each, fresh independent draws — including parity shares the proactive path never sent). That is ~5× more recoveries per budget byte than resending full frames. NACKs fire only while deadline − now > 1.15·est_RTT + 4ms (RTT learned live as an EWMA), are spaced, capped, and token-bucketed; retransmits stop at 1.95× total so ARQ can never starve the proactive stream.

5. System-level details

  • Budget safety. A running byte counter hard-clamps proactive output at 1.99×·frames·160 (structurally it sits at 1.881×) and ARQ output at 1.95× total; feedback is ≤ 60 NACKs/s × 6 B < 0.05×. Worst-case sum stays < 2.0×.
  • Buffer ownership. Sender: fixed 4096-frame payload ring (answers NACKs by re-encoding shares on demand). Receiver: 1024-slot reassembly ring (seq-tagged share buffers + bitmask) plus delivered/NACK bitsets. No allocation on the data path.
  • Byte order. Big-endian on every wire field, converted at the edges.
  • Never block. Both loops are poll()-driven with short ticks so scheduled shares (burst spreading) and NACK bookkeeping run even when nothing arrives.
  • Knobs, one binary. FLAKY_RS_P (parities), FLAKY_RS_SPREAD (spread shares over 1–3 ticks against burst loss), NACK timing/tries — all env-tunable, defaults are the graded configuration (P=3, spread=1, ARQ on).

6. What breaks it, honestly

Sustained heavy burst loss (Gilbert-Elliott bursts eating ~15% of packets in clumps, plus delay spikes) still exceeds 1% at any delay within the 2× budget: bursts are per-packet on the lane, so they hit whatever you send, and the in-time recovery paths share them. Share-spreading over 2 ticks plus mask-ARQ cut that synthetic profile's misses from 5–9% to ~2.5%, but under 1% needs more than 2× — a budget ceiling, not a delay ceiling. On loss/jitter/reorder/dup regimes like the practice profiles, the design is valid below the network's max jitter — a statistical floor no copy-based scheme reaches.
